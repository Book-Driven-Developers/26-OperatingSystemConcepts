## 운영체제

<aside>
❓

컴퓨터 시스템의 자원(Resource)을 관리하는 시스템 관점(System View)에서 볼 때, 운영체제의 가장 중요한 역할은 무엇인가?

</aside>

- 시스템 관점(System View) 에서 운영체제는 자원 할당자(Resource Allocator)이자 제어 프로그램으로서 CPU, 메모리, 입출력 장치 등의 리소스를 효율적이고 공정하게 관리하는 역할을 한다.

## 컴퓨터 시스템

<aside>
❓

컴퓨터를 부팅하면 어떤 과정을 거쳐 운영체제가 실행되는가?

</aside>

1. 시작
2. **부트스트랩 프로그램 실행**
    1. 시스템 초기화
    2. **부트로더 프로그램 실행**
        1. **운영체제 실행 (커널이 메모리에 로드됨)**

## 시분할 시스템 (멀티 태스킹 시스템)

<aside>
❓

CPU는 어떻게 작동하길래 사용자가 여러 프로그램이 마치 '동시에' 실행되는 것처럼 느끼고 빠른 응답을 받을 수 있을까? (시분할 시스템에서)

</aside>

- 운영체제가 여러개의 프로세스를 빠르게 전환하며 실행함.
    - 사용자는 여러 프로그램이 마치 동시에 실행되는 것처럼 느끼게 됨.
    - 여러개의 프로세스는 동시에 메모리에서 관리됨.
- 멀티태스킹 시스템에서는 반응 시간(Response Time)을 줄이는 게 중요하다.

### 멀티프로그래밍

- 여러 프로그램을 메모리에 로드하고, 한 프로세스가 대기 상태가 되면 다른 프로세스의 작업을 수행한다.

## 인터럽트

<aside>
❓

트랩과 인터럽트의 차이는 무엇인가?

</aside>

**트랩**

- 소프트웨어에 의해 발생하는 인터럽트는 트랩(Trap)이라고 부른다.
- 시스템 콜(System call)이라는 특별한 명령으로 인터럽트를 발생시킨다.

**인터럽트**

- 시스템 버스(System bus)를 통해 CPU에 신호를 보냄으로써 인터럽트를 발생시킨다.
- 신호를 받은 CPU는 하던 일을 멈추고 받은 인터럽트를 처리한다.

<aside>
❓

인터럽트를 받은 CPU가 원래 하던 일을 멈추고 인터럽트를 처리한 뒤, **다시 원래 실행하던 프로그램으로 문제없이 돌아가기 위해 운영체제는 내부적으로 어떤 과정(또는 상태 저장)들을 거쳐야 할까?**

</aside>

1. 프로그램 카운터와 CPU 레지스터 값을 PCB/시스템 스택에 백업한다.
2. 인터럽트 벡터에 접근한다.
    1. 인터럽트 벡터에는 서비스 루틴의 주소가 저장되어 있다.
3. 인터럽트 서비스 루틴이 실행된다.
    1. 서비스 루틴에는 하드웨어적 작업(읽기/쓰기)이 하드코딩되어 있음.

## DMA

<aside>
❓

하드 디스크에서 메모리로 대용량 파일을 복사할 때 1바이트 단위로 매번 인터럽트가 발생한다면, CPU는 계속 하던 일을 멈춰야 해서 효율이 크게 떨어질 것이다.

**이러한 병목 현상을 해결하기 위해 도입된 DMA(Direct Memory Access) 장치는 어떤 방식으로 동작하며, 결과적으로 CPU의 개입을 어떻게 줄여줄까?**

</aside>

- CPU를 거치지 않고 DMA 컨트롤러가 직접 장치의 데이터를 메모리에 전달한다.
- 데이터 이동이 완료되면 CPU에게 완료 인터럽트를 한번 보낸다.

## 커널

- 운영체제의 핵심으로, 커널과 커널 모듈로 이루어져 있다.
- 커널이 같다면 같은 운영체제로 취급한다.

### 커널 모드와 유저 모드

<aside>
❓

잘못 작성된(또는 악의적인) 프로그램이 운영체제 코드를 마음대로 건드리거나 시스템을 망가뜨리려고 한다면 어떻게 막을 수 있을까?

</aside>

위험으로부터 시스템을 보호하기 위해 사용자 프로그램이 직접 운영체제의 핵심 기능에 접근하지 못하도록 모드를 분리함.

**커널 모드**

- HW 모드 비트 = 0
- 인터럽트/트랩 발생 시 커널 모드로 변경됨.
- privileged instruction은 오직 커널 모드에서만 실행됨.

**유저 모드**

- HW 모드 비트 = 1
- **`시스템 콜 함수`**를 호출하면 커널 모드로 변경됨.

## 타이머 (타이머)

<aside>
❓

프로그램이 무한 루프에 빠져 시스템 콜도 전혀 호출하지 않고, 오류도 내지 않은 채 혼자서 계속 CPU를 차지하고 있을때, **운영체제는 어떻게 프로그램으로부터 CPU 제어권을 다시 빼앗아 올 수 있을까?**

</aside>

- 운영체제는 클록이 매 tick 할 때마다 감소하는 Counter를 설정함.
- Counter가 0이 되면, **`타이머 인터럽트`**가 발생함.
    - CPU 제어권이 운영체제로 넘어옴.

## 프로그램과 프로세스

<aside>
❓

'프로그램'과 '프로세스'는 각각 어떤 차이가 있는가?

</aside>

**프로그램**

- 디스크에 저장되어 있음.
- 마치 파일처럼 아무런 동작을 하지 않음.

**프로세스**

- 메모리에 적재되어 자원(CPU, 메모리…)를 할당받아 순차적으로 실행됨.
- **`프로그램 카운터`**를 가짐.
    - 프로그램 카운터는 다음 실행할 명령어를 가리킴.
    - 쉽게 말해, 북마크와 같다.
- 두 개의 프로세스가 동일한 프로그램이라 하더라도, 별도의 수행 순서를 가진 독립적인 개체.
- 싱글 스레드는 프로그램 카운터를 1개, 멀티 스레드는 여러개의 프로그램 카운터를 가지고 있다.

## 캐시

### 캐시 일관성 문제

<aside>
❓

디스크에 있는 변수 A의 값을 1 증가시키기 위해 '메인 메모리 ➡️ 캐시 ➡️ 내부 레지스터' 순서로 데이터를 복사해 온다. 다중 태스킹이나 다중 처리기 환경에서는 이 과정에서 **`캐시 일관성`** 문제가 발생하는데, **`캐시 일관성`** 문제란 무엇인가?

</aside>

- 데이터가 디스크 → 메인 메모리 → 캐시 → 내부 레지스터로, 여러 저장장치에 복사본으로 존재 함.
    - 이때 내부 레지스터에서 값을 변경하면, 하드디스크에 저장하기 전까지 저장 장치마다 데이터 값이 서로 다른 상태가 되는 문제가 발생함.
- 단일 프로세스 환경에서는 가장 최신 저장장치의 값을 읽어오면 됨.
- 다중 프로세스 환경에서는, 모든 프로세스/CPU가 가장 최근에 갱신된 최신 값을 얻도록 보장해야 함.
    - 다중 태스킹 환경 → 여러 개의 프로세스가 번갈아 실행됨.
    - 다중 처리기 환경 → 각자의 로컬 캐시를 가진 CPU가 동시에 실행됨.
- 분산 환경에서는, 복제본을 한 곳에서 업데이트 하면 다른 모든 복제본을 가능한 빨리 최신 상태로 유지하는 것을 보장함.
