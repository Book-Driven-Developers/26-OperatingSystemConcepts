## 4.1 스레드

- 프로세스의 작업 흐름

- 싱글 스레드
    - 하나의 프로세스가 **한 번에 하나의 작업만 수행**
- 멀티 스레드
    - 하나의 프로세스가 **동시에 여러 작업을 수행**

### 여러개의 프로세스보다 스레드를 두는 이유

1. 프로세스끼리는 데이터를 공유하기 어렵다.
    - 두 프로세스가 하나의 데이터를 공유하려면 메시지 패싱이나 공유 메모리 또는 파이프를 사용해야 하는데, 효율도 떨어지고 개발자가 구현, 관리하기도 번거롭다.
2. 프로세스 생성 작업은 많은 시간과 자원을 필요로 한다.
3. 프로세스 사이 컨텍스트 스위치가 계속 일어나면 성능 저하가 발생한다. 
    - 스레드 전환에도 컨텍스트 스위치가 일어나지만 속도가 더 빠르다.

### 스레드의 고유 자원과 공유 자원

고유 자원

- 스레드 ID
- 프로그램 카운터
- 레지스터 집합
- 스택
    - 함수 호출
    - 지역 변수 저장

스레드가 자신의 '실행 흐름'을 관리하기 위해 필요한 정보들

공유 자원

- 코드
- 데이터 섹션
- 열린 파일
- 신호

### 다중 스레드 프로그래밍 장점

장점

1. 응답성.
    - 요청 작업이 오래 걸리는 작업이라 해도, 다른 스레드가 대신 응답을 처리할 수 있음
2. 자원 공유.
    - 프로세스 내의 코드와 데이터를 공유함
3. 경제성. 
    - 프로세스를 새로 만들고 전환하는 것보다 비용(시간, 메모리)이 훨씬 저렴
4. 규모 적응성. 
    - 다중 코어 환경에서, 각각의 스레드를 여러 처리기에 분산시켜 병렬로 실행 가능.

## 4.2 다중 코어 프로그래밍

### Concurrency와 Parallelism

동시성(Concurrency)

- 싱글 프로세서 시스템에서 사용
- 프로세서가 여러 개의 스레드를 번갈아가며 수행
- **동시에 실행되는 것처럼 보이게 하는 방식**

병렬성(Parallelism)

- 멀티코어 시스템에서 사용
- 여러 개의 코어가 각 **스레드를 동시에 수행하는 방식**

## 4.3 다중 스레드 모델

### 유저 스레드와 커널 스레드

유저 스레드

- 사용자 수준의 스레드 라이브러리가 관리하는 스레드
- 안정성은 떨어지지만 성능이 저하되지는 않는다.
- 대표적으로 POSIX Pthreads, Win32 threads, Java threads가 있음.

커널 스레드

- 커널이 관리하는 스레드
- 안정적이다.
    - 라이브러리를 위한 코드와 자료구조가 커널 공간에 있다.
- 유저 모드에서 커널 모드로 계속 바꿔줘야 한다.
    - 라이브러리 API 호출 → 커널 시스템콜 호출
    - 잦은 호출 시 성능 문제 발생 가능

### 다대일 모델 (Many To One Model)

- 하나의 커널 스레드에 여러 개의 유저 스레드를 연결하는 모델
- **한 번에 하나의 유저 스레드만 커널에 접근 가능**
    - 한 스레드가 Blocking 시스템 콜 할 경우 전체 프로세스가 Blocking 됨
- 멀티코어 시스템에서 병렬적인 수행을 할 수가 없다.

→ 요즘에는 잘 사용되지 않는 방식

### 일대일 모델 (One To One Model)

- 하나의 유저 스레드에 하나의 커널 스레드가 대응하는 모델
- 동시성을 높여주고, 멀티프로세서 시스템에서는 동시에 여러 스레드를 수행할 수 있다. (병렬성 높음)
- 사용자 스레드를 생성할 때마다 무거운 커널 스레드도 함께 생성해야 함.
    - 커널 스레드를 생성하는 것은 오버헤드가 큰 작업이기 때문에 성능 저하가 발생할 수 있다.

### 다대다 모델 (Many To Many Model)

- 여러 유저 스레드에 더 적거나 같은 수의 커널 스레드가 대응하는 모델
- 필요한 만큼 사용자 스레드 생성 → 상응하는 커널 스레드가 다중 처리기에서 병렬로 실행

단점

- 좋아보이지만 구현이 어렵다.
- 처리 코어 수가 증가함에 따라 커널 스레드 수가 제한이 중요해지고 있음.

### 두 수준 모델

- Many-to-Many 모델과 비슷
- 특정 **유저 스레드를 위한 커널 스레드를 따로 제공**하는 모델
    - 점유율이 높아야 하는 유저 스레드를 더 빠르게 처리해줄 수 있다.

## 4.4 스레드 라이브러리

### 동기 스레딩과 비동기 스레딩

동기 스레딩 (= fork-join 전략)

- 부모 스레드가 자식 스레드를 생성(fork)하고 자식이 모두 종료할 때까지 기다렸다가(join) 자신의 실행을 재개하는 방식
- 부모가 생성한 자식들끼리는 concurrently하게 실행됨.

비동기 스레딩

- 부모 스레드가 자식 스레드를 생성한 후, 부모는 자신의 실행을 재개하여 두 스레드를 병행하게 실행
- 부모는 자식의 종료를 알 필요 없음. 부모와 자식은 독립적으로 실행함.

### Java 스레드

Runnable

```java
public class MyRunnable implements Runnable {
    @Override
    public void run() {
        // 스레드가 실행할 작업을 여기에 구현
    }

    public static void main(String[] args) {
        MyRunnable myRunnable = new MyRunnable();
        Thread thread = new Thread(myRunnable);
        thread.start(); // 스레드 시작
    }
}
```

- Java 에서 멀티스레드를 구현하기 위해 사용되는 인터페이스
- Runnable 인터페이스를 구현하는 객체는 `run()` 메서드를 구현해야 함

Callable

```java
import java.util.concurrent.*;

public class MyCallable implements Callable<Integer> {
    @Override
    public Integer call() throws Exception {
        // 스레드가 실행할 작업을 여기에 구현
        return 42; // 예시로 정수 42를 반환
    }

    public static void main(String[] args) throws ExecutionException, InterruptedException {
        MyCallable myCallable = new MyCallable();
        ExecutorService executorService = Executors.newSingleThreadExecutor();
        Future<Integer> future = executorService.submit(myCallable);

        // 작업 완료 대기하고 결과 값 가져오기
        int result = future.get();
        System.out.println("결과: " + result);

        executorService.shutdown();
    }
}
```

- JDK 1. 5 부터 도입된 인터페이스
- 작업의 결과를 반환할 수 있다.
    - 전역 변수 개념이 없는 Java에서, 결과를 반환하는 스레드로서 동작함.
- `ExecutorSevice`를 사용하여 스레드를 실행함
    - ExecutorService에 Task만 지정해주면 알아서 ThreadPool을 이용해서 Task를 실행하고 관리한다.

Future 객체

1. `Callable` 작업이 스레드 풀에 제출
2. 결과를 `Future` 객체 형태로 반환
3. 부모 스레드는 나중에 `Future` 인터페이스의 `get()` 메소드를 호출하여 결과 확인 가능

→ 스레드의 실행과 결과를 분리

## 4.5 암묵적 스레딩

### 스레드 풀

- 스레드를 요청할 때마다 매번 새로운 스레드를 생성하고, 수행하고, 지우고를 반복하면 성능이 저하된다.
- **미리 스레드 풀에 여러 개의 스레드를 만들어두고** 요청이 오면 스레드 풀에서 스레드를 할당해주는 방법을 사용

장점

1. 새 Thread를 만들어 주는 것보다 생성된 Thread를 사용하여 서비스해 주는 것이 더 빠르다.
2. Thread pool은 임의의 시간에 존재할 수 있는 Thread 개수에 제한을 둔다.
3. task를 생성하는 방법을 분리하여, task를 일정 시간 후에 실행되도록 하거나, 주기적으로 실행시킬 수 있다.

## 4.6 스레드와 관련된 문제들

### Fork() 및 Exec() 시스템 콜

<aside>
❓

만일, 한 프로그램의 Thread가 fork()를 호출하면, 새로운 프로세스는 모든 Thread를 복제해야 하는가?

</aside>

- `exec()`이 곧바로 모든 것을 새로운 프로그램으로 덮어씌울 예정이라면, 호출한 스레드 하나만 복사.
- 반면 다른 프로그램을 실행하지 않고 기존 작업을 그대로 이어갈 거라면 모든 스레드를 복사해야 함.

### 스레드 취소

- Thread가 끝나기 전에 그것을 강제 종료시키는 작업
    - ex) 여러 Thread들이 데이터베이스를 병렬로 검색하고 있다가 한 Thread가 원하는 결과를 찾게 되면 나머지 Thread들은 취소되어도 된다.

목적 스레드

- 취소되어할 Thread들을 목적 스레드(Target Thread)라고 부른다.

비동기식 취소

- 한 Thread가 즉시 Target Thread를 강제 종료시킨다.
- 목적 스레드의 작업과 관계없이 즉시 종료됨.
    - 공유 자원 갱신 등의 작업일 경우 문제가 됨.

지연 취소

- Target Thread가 주기적으로 자신이 강제 종료되어야 할지 점검한다.
- 자신이 취소되어도 안전하다고 판단될 때 취소 여부를 검사할 수 있음.
