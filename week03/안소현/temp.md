## 1. Thread

- CPU 사용의 기본 단위 
- 각 스레드는 독립적으로 실행되며, 프로세스 내부에서 다른 스레드와 자원을 공유하면서 작업을 수행한다.

### 구성요소 

각 스레드는 아래의 정보를 독립적으로 가진다. 
- Thread ID: 운영체제가 각 스레드를 식별하기 위한 고유한 ID
- Program Counter (PC) : 현재 실행 중인 명령어의 주소를 저장한다.
- Register Set : CPU 레지스터 값을 저장한다.
- Stack : 함수 호출, 지역 변수 등을 저장하는 메모리 영역이다.

같은 프로세스에 속한 스레드는 다음 자원을 공유한다.
- Code Section
- Data Section
- Heap
- Open Files
- Signals

공유 자원을 통해 스레드 간 데이터 교환이 쉽지만, 동시에 여러 스레드가 접근할 경우 동기화 문제가 발생할 수 있다.

## 2. Multi-Threading

- 하나의 프로세스에서 여러 개의 스레드가 동시에 실행되는 방식

### multi processingr과 비교

| 구분 | 멀티 프로세스 | 멀티 스레드 |
|------|--------------|-------------|
| 실행 단위 | 프로세스 | 스레드 |
| 메모리 | 독립적인 메모리 공간 사용 | 프로세스의 메모리 공간 공유 |
| 자원 공유 | IPC(Inter-Process Communication) 필요 | 메모리 직접 공유 |
| 생성 비용 | 상대적으로 큼 | 상대적으로 작음 |
| Context Switch | 비용이 큼 | 비용이 작음 |
| 안정성 | 하나의 프로세스 오류가 다른 프로세스에 영향 없음 | 하나의 스레드 오류가 전체 프로세스에 영향을 줄 수 있음 |

### multi-threading 장점
1. Responsiveness (응답성)
프로세스의 일부가 오래 걸리는 작업을 수행하더라도 다른 스레드는 계속 실행될 수 있다.
- 예를 들어 웹 브라우저는 파일을 다운로드하는 동안에도 화면을 조작하거나 스크롤할 수 있다.

2. Resource Sharing (자원 공유)
같은 프로세스에 속한 스레드는 메모리와 자원을 공유한다. 따라서 데이터를 공유하기 쉽고, 프로세스 간 통신(IPC)에 비해 효율적이다.

3. Economy (경제성)
스레드를 생성하고 종료하는 비용은 프로세스보다 훨씬 작다. 또한 Context Switch 시 공유 자원을 그대로 사용할 수 있으므로 전환 비용도 감소한다.

4. Scalability (확장성)
티코어 시스템에서는 여러 스레드를 서로 다른 CPU 코어에서 동시에 실행할 수 있어 시스템 성능을 향상시킬 수 있다.

### multi-threading 단점
1. 동기화(Synchronization)
스레드가 메모리를 공유하므로 동시에 같은 데이터를 수정하면 Race Condition이 발생할 수 있다. 이를 방지하기 위해 Mutex, Semaphore 등의 동기화 기법이 필요하다.

2. 디버깅의 어려움
여러 스레드가 동시에 실행되기 때문에 실행 순서가 항상 동일하지 않아 재현이 어려운 버그가 발생할 수 있다.

3. 안정성
스레드는 같은 프로세스의 주소 공간을 공유한다. 따라서 하나의 스레드에서 잘못된 메모리 접근이나 치명적인 오류가 발생하면 프로세스 전체가 종료될 수 있다.


## 3. Parallelism vs Concurrency

### Concurrency
여러 작업을 동시에 수행하는 것처럼 보이도록 작업을 번갈아 실행하는 것
- 단일 CPU 환경에서는 운영체제가 빠르게 스레드를 전환하여 여러 작업이 동시에 수행되는 것처럼 보이게 한다. 

### Parallelism
여러 작업을 실제로 동시에 실행하는 것
- 멀티 코어 환경에서는 여러 CPU 코어가 각각 다른 작업을 실제로 동시에 수행할 수 있다. 


## 4. 암달의 법칙 

프로그램의 일부를 병렬화했을 때, 전체 성능이 얼마나 향상될 수 있는지 계산하는 법칙

- 프로그램 전체가 병렬로 실행될 수 있는 것은 아니며, 반드시 순차적으로 실행해야하는 부분이 존재하기 때문에 CPU 코어나 스레드 수를 계속 늘리더라도 전체 성능 향상에는 한계가 있다. 
- 적용 가능한 상황
  - 멀티 코어 도입 효과를 계산할 때,
  - 멀티 스레드 프로그램의 성능 한계를 분석할 때,
  - 최적화 우선 순위를 결정할 때,
  - 하드웨어 확장의 효과를 판단할 때,

## 5. 다중 스레드 모델 

운영체제는 사용자 스레드(User Thread)와 커널 스레드(Kernel Thread)를 다양한 방식으로 연결하여 관리한다.

- 사용자 스레드(User Thread)
  - 사용자 영역(User Space)의 스레드 라이브러리에서 관리하는 스레드
  - 스레드의 생성, 종료, 스케줄링 등을 사용자 영역에서 수행하므로 관리 비용이 작다.

- 커널 스레드(Kernel Thread)
  - 운영체제가 직접 관리하는 스레드
  - 커널이 스케줄링을 수행하며, 실제 CPU에서 실행되는 단위이다.

### Many-to-One 모델

여러 개의 사용자 스레드를 하나의 커널 스레드에 매핑

- 스레드 관리 비용이 작다.
- 사용자 영역에서 스레드를 빠르게 생성할 수 있다.
- 하나의 스레드가 시스템 콜 등으로 Block되면 모든 스레드가 함께 Block된다.
- 멀티코어를 활용한 병렬 실행이 불가능하다.

### One-to-One 모델

각 사용자 스레드가 하나의 커널 스레드와 연결

- 하나의 스레드가 Block되어도 다른 스레드는 계속 실행될 수 있다.
- 멀티코어에서 병렬 실행이 가능하다.
- 커널 스레드를 많이 생성해야 하므로 오버헤드가 증가한다.

### Many-to-Many 모델

여러 사용자 스레드를 여러 커널 스레드에 동적으로 매핑

- 사용자 스레드 수보다 적은 수의 커널 스레드를 사용할 수 있다.
- Block되는 스레드가 있어도 다른 스레드는 계속 실행된다.
- 멀티코어 활용이 가능하다.
- 사용자 스레드와 커널 스레드 수를 유연하게 조절할 수 있다.

### 두 수준 모델(Two-Level Model)

Many-to-Many 모델을 확장한 형태, 필요한 경우 특정 사용자 스레드를 특정 커널 스레드에 직접 바인딩 할 수 있다.

### 정리

| 모델 | 특징 | 멀티코어 활용 | Block 영향 |
|------|------|--------------|------------|
| Many-to-One | 여러 User Thread → 하나의 Kernel Thread | 못함 | 모든 스레드 Block |
| One-to-One | User Thread 하나당 Kernel Thread 하나 | 가능 | 해당 스레드만 Block |
| Many-to-Many | 여러 User Thread ↔ 여러 Kernel Thread | 가능 | 다른 스레드 계속 실행 |
| Two-Level | Many-to-Many + 특정 스레드 Binding | 가능 | 다른 스레드 계속 실행 |


## 6. 비동기 스레딩 vs 동기 스레딩

- 비동기 스레딩
  - 부모 스레드가 새로운 스레드를 생성한 후 자신의 작업을 계속 수행하는 방식
  - 생성된 자식 스레드의 종료를 기다리지 않으며, 부모와 자식 스레드가 독립적으로 실행된다.
- 동기 스레딩
  - 부모 스레드가 새로운 스레드를 생성한 후 자식 스레드의 작업이 끝날 때까지 기다리는 방식
  - `join()`을 통해 자식 스레드의 종료를 기다리는 방식이 해당된다.


## 7. Java Thread

Java는 스레드를 생성하기 위해 `Thread` 클래스와 `Runnable` 인터페이스를 제공한다.

Java에서 스레드를 생성하는 방법은 크게 두 가지이다.
1. Thread 클래스 상속 : `Thread` 클래스를 상속받아 `run()` 메서드를 구현한다.

```java
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Thread Running");
    }
}

new MyThread().start();
```

2. Runnable 인터페이스 구현

`Runnable` 인터페이스의 `run()` 메서드를 구현한 뒤 `Thread` 객체에 전달한다.

Runnable 방식은 작업(Task)과 실행(Thread)을 분리할 수 있으며, Java의 단일 상속(Single Inheritance) 제약도 피할 수 있다.

```java
class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("Task Running");
    }
}

Thread thread = new Thread(new MyTask());
thread.start();
```

### Thread 실행 과정

Java에서 스레드는 `run()`이 아닌 `start()`를 호출해야 한다.

```java
Thread thread = new Thread(new MyTask());

thread.start();
```
 `run()`을 직접 호출하면 새로운 스레드가 생성되지 않고 일반 메서드 호출처럼 현재 스레드에서 실행된다.

### Executor Framework

직접 `Thread`를 생성하기보다 Executor Framework를 사용하는 것이 권장된다.

Executor Framework는 스레드 생성과 관리를 대신 수행하며, 개발자는 실행할 작업(Task)만 제출하면 된다.
Executor Framework는 내부적으로 Thread Pool을 사용하여 여러 작업을 관리한다.

```java
ExecutorService executor = Executors.newFixedThreadPool(4);

executor.execute(new MyTask());
```

#### 스레드 풀 종류

ThreadPoolExecutor : 일반적인 독립적인 작업(Task) 처리
- 하나의 공유 작업 큐 사용
- `newFixedThreadPool(n)`: 고정된 수의 스레드로 작업을 처리
- `newCachedThreadPool()`: 필요에 따라 스레드를 생성하고 유휴 스레드를 재사용
- `newSingleThreadExecutor()`: 하나의 스레드에서 작업을 순차적으로 실행
- `newScheduledThreadPool(n)`: 작업을 지연 실행하거나 주기적으로 실행

ForkJoinPool : 큰 작업을 여러 개의 작은 작업으로 분할하여 병렬 처리
- 각 스레드가 자신의 작업 큐를 가짐
- `newWorkStealingPool()`: 작업이 없는 스레드가 다른 스레드의 작업을 가져와 처리

Java 21+
- `newVirtualThreadPerTaskExecutor()`: 각 작업을 하나의 가상 스레드에서 실행한다.


## 8. Thread issues

### 신호 처리(Signal Handling)

Signal은 프로세스에 특정 사건이 발생했음을 알리기 위해 UNIX 계열 운영체제에서 사용하는 통지 방식이다.

대표적인 사건은 다음과 같다.
* 잘못된 메모리 접근
* 0으로 나누기
* 타이머 만료
* 사용자 인터럽트
* 다른 프로세스가 보낸 종료 요청

신호는 발생 원인에 따라 동기 신호와 비동기 신호로 구분할 수 있다.
- 동기 신호 : 그 신호를 야기한 스레드에 전달되어야 하고, 다른 스레드에는 전달되면 안된다. 
- 비동기 신호 : 정해져 있지 않다. (그 프로세스의 모든 스레드에게 전달해야할 수도 있고, 그냥  그 신호를 무시하지 않는 스레드 중 첫번째에게만 전달할 수도 있다.)


### 스레드 취소

스레드 취소는 스레드가 작업을 끝내기 전에 실행을 종료시키는 것을 의미한다.

- 비동기 취소 : 한 스레드가 목적 스레드를 즉시 강제 종료시키는 방식
	```text
	Thread A ── cancel ──▶ Thread B 즉시 종료
	```
	- 문제점
    	- 대상 스레드가 어떤 작업을 수행하고 있는지 고려하지 않고 즉시 종료된다.
    	- 예를 들어 Lock을 보유한 스레드가 즉시 종료되면 Lock이 해제되지 않아 다른 스레드가 무한히 대기할 수 있다.
	- 비동기 취소는 즉시 종료된다는 장점이 있지만, 공유 자원과 프로그램 상태를 일관되게 유지하기 어렵다.
- 지연 취소 : 대상 스레드가 자신이 안전하게 종료될 수 있는 시점에 취소 요청을 확인하고 스스로 종료하는 방식
	```text
	취소 요청 발생 ──▶ Target Thread 계속 실행 ──▶ Cancellation Point 도달 ──▶ 취소 요청 확인 후 종료
	```
	- 대상 스레드는 취소 요청이 있었는지를 주기적으로 확인한다. 취소 여부를 확인하는 지점을 `Cancellation Point`라고 한다. Cancellation Point는 다음과 같이 스레드가 안전하게 종료될 수 있는 위치에 둔다.
    	* 반복문의 특정 지점
    	* 대기 함수 호출 시점
        * 블로킹 시스템 콜 호출 시점
        * 공유 자원 정리가 끝난 시점
	- 따라서 비동기 취소보다 안전하게 스레드를 종료할 수 있다.

### 스레드 로컬 저장소(Thread-Local Storage)

같은 프로세스의 스레드는 일반적으로 프로세스의 데이터를 공유한다. 하지만 경우에 따라 각 스레드가 자신만의 데이터를 가져야 할 때, 사용하는 것이 이때 사용하는 것이 **Thread-Local Storage, TLS**이다.

- 지역 변수와의 차이

일반 지역 변수도 각 스레드의 Stack에 저장되므로 스레드마다 독립적이다. 하지만 지역 변수는 하나의 함수 호출 범위에서만 사용할 수 있다. 반면 TLS는 스레드가 실행되는 동안 여러 함수에서 접근해야 하는 스레드 전용 데이터를 저장하는 데 사용된다.

TLS는 다음과 같은 경우에 유용하다.
* 각 스레드가 고유한 식별자를 가져야 하는 경우
* 스레드마다 별도의 통계 정보를 저장하는 경우
* 스레드마다 독립적인 난수 생성 상태를 유지하는 경우
* 스레드 풀의 각 워커가 고유한 작업 정보를 유지하는 경우
* 함수에 매번 데이터를 인자로 전달하기 어려운 경우


### 스케줄러 액티베이션(Scheduler Activations)

스케줄러 액티베이션은 사용자 수준 스레드 라이브러리와 커널이 서로 협력하여 스레드를 스케줄링하는 방법

특히 Many-to-Many 모델과 Two-Level 모델에서 중요하다.

이 모델에서는 여러 사용자 스레드가 여러 커널 스레드에 매핑된다.
사용자 스레드는 사용자 영역의 스레드 라이브러리가 관리하고, 커널은 커널 스레드를 스케줄링한다.

따라서 사용자 스레드 라이브러리와 커널 사이에 정보를 교환할 방법이 필요하다.

#### LWP(Lightweight Process)

Many-to-Many 모델에서 사용자 스레드와 커널 스레드 사이에는 **LWP**라는 중간 자료구조가 사용될 수 있다. 사용자 스레드 라이브러리의 관점에서 LWP는 사용자 스레드를 실행할 수 있는 가상 프로세서처럼 동작한다.

- 각 LWP는 하나의 커널 스레드에 연결 ──▶ 사용자 스레드를 실행하려면 사용 가능한 LWP가 필요
- 스케줄러 액티베이션의 목적 : User Thread가 Block되어도 CPU가 놀지 않도록, 커널이 Thread Library에 알려주고 필요한 만큼 LWP를 동적으로 제공하는 메커니즘

LWP로 가능한 것
* Blocking된 스레드 때문에 전체 사용자 스레드가 멈추는 문제 방지
* 실행 가능한 사용자 스레드 수에 맞춰 커널 스레드 조절
* 사용자 수준 스케줄링과 커널 수준 스케줄링의 협력
* Many-to-Many 모델의 효율적인 구현
