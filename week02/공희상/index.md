# 프로세스

## 프로세스 정의

운영체제에서 프로그램을 실행하는 작업 단위를 의미한다.

## 프로세스 구조

code(text) : 컴파일된 소스 코드
Data Section : 전역 변수
Heap Section : 런타임 과정에서 동적으로 할당되는 변수
Stack Section : 함수가 실행되는 동안 지역변수가 임시로 저장

## 프로세스 상태

![img_1.png](img_1.png)

- 생성(new): 프로세스 생성 상태
- 준비(ready): 프로세서에 의해 실행되기를 대기하는 상태
- 실행(running): 프로세스가 자원을 할당받아 수행되는 상태
- 대기(waiting): 프로세스 이벤트가 발생되어 대기하는 상태
- 종료(terminated): 프로세스 실행 종료 상태

## 프로세스 제어 블록(PCB, Process Control Block)

PCB는 프로세스에 대한 정보 블록으로, 운영체제가 프로세스를 관리하기 위해 사용하는 자료구조이다. PCB에는 다음과 같은 정보가 포함된다.

- 프로세스 상태(Process State): 현재 [프로세스의 상태](#프로세스-상태)
- 프로그램 카운터(Program Counter): 이어서 실행해야 할 다음 명령어 주소
- CPU 레지스터(CPU registers): 프로세스가 올바르게 재실행하기 위한 PC와 여러 범용 레지스터 등의 값
- CPU 스케줄링 정보(CPU-scheduling information): 프로세스의 중요도, 스케줄링 큐 포인터 등 프로세스 실행 순서를 정하는 정보
- 메모리 관리 정보(Memory-management information): base, limit 레지스터 값, 페이지 테이블 등 메모리 시스템 정보
- 통계 정보(Accounting Information): 프로세스의 개수, 타임아웃 정보, 실행 ID 등에 사용되는 CPU양의 정보
- 입/출력 상태 정보(I/O status information): 프로세스에 할당된 입출력 장치 목록

# 프로세스 스케줄링

- 멀티프로그래밍(Multi-programming)의 목적은 CPU 효율 극대화를 위해 동시에 여러 프로세스를 실행하는 것
- 시분할 시스템(Time Sharing System)의 목적은 CPU 코어가 빈번하게 프로세스들 사이에서 변경하여 사용자가 각각의 프로그램을 동시에 수행되는 것처럼 하기 위함

## 문맥 교환(Context Switch)

- 프로세스의 Context는 실행을 이어가기 위해 필요한 상태 정보로, PCB에 저장
- 인터럽트가 발생하면 커널은 현재 실행 상태를 저장하여, 인터럽트 처리가 끝난 뒤 기존 작업을 계속할 수 있게 함
- CPU가 현재 프로세스에서 다른 프로세스로 실행 대상을 변경하는 것을 컨텍스트 스위칭
- 컨텍스트 스위칭 중에는 사용자 프로그램의 실제 작업이 진행되지 않으므로, 소요되는 시간은 오버헤드에 해당한다.

# 프로세스에 대한 연산

## 프로세스 생성

- 프로세스는 자식 프로세스를 생성할 수 있으며, 트리 구조를 형성한다.
- 각 프로세스들은 pid로 식별되며 고유한 값을 가진다.
- fork(): 부모 프로세스를 복제하여 자식 프로세스 생성
- exec(): 새 프로그램을 자신 프로세스의 주소 공간에 불러온다. 
- wait(): 자식 프로세스가 종료될 때까지 부모 프로세스를 block 시킨다.
- fork()의 반환값: 부모에게는 자식의 pid, 자식 프로세스에게는 0이 반환된다.

## 프로세스 종료(Process Termination)

- 마지막 명령어 실행 후 `exit()`으로 프로세스를 종료 요청 → 부모에게 `wait()` 시스템 콜을 통해 status를 받는다.
- 부모 프로세스가 자식 프로세스를 종료시키는 경우
  - 자식이 자신에게 할당된 자원을 초과하여 사용하는 경우
  - 자식에게 할당된 task가 더 이상 필요가 없는 경우
  - 부모가 종료되었고, 운영체제가 그 프로세스의 자식이 실행하는 것을 허용하지 않는 경우
- 좀비(zombie): 자식은 종료됐지만 부모가 아직 wait()를 호출하지 않은 상태 
- 고아(orphan): 부모가 wait() 없이 먼저 종료된 경우 → init/systemd가 새 부모가 되어 수거

# 프로세스 간 통신(Interprocess Communication)

협력하는 프로세스가 통신하는 이유
- 정보 공유(Information Sharing): 여러 사용자가 동시에 동일한 정보에 대해 접근하려고 할 수 있다.
- 계산 가속화(Computation Speed-up): 특정 task를 빨리 실행시키기 위해, task를 sub-task들로 나누어 병렬로 실행되게 한다.
- 모듈성(Modularity): 시스템 기능을 별도의 프로세스들(Thread들)로 나누어, 모듈식 형태로 구성하기를 원할 수 있다.
- 편의성(Convenience): 한 사용자가 동시에 작업할 다수의 task들을 가질 수도 있다.

프로세스들은 데이터와 정보를 공유하기 위해 IPC(Inter-process communication, 프로세스 간 통신) 기법이 필요하다.

IPC에는 공유 메모리(Shared Memory)와 메시지 전달(Message Passing) 모델이 있다.
## 공유 메모리 (Shared Memory)
- 두 프로세스가 공유 메모리 영역을 설정하고 직접 읽고 씀
- 대표 예: **생산자-소비자 문제** (bounded buffer)
- 커널 개입이 적어 **빠름**, 대신 동기화는 프로그래머 책임

## 메시지 전달 (Message Passing)
- `send()` / `receive()` 연산으로 통신
- **버퍼링(큐) 방식 3가지**
  - zero capacity: 보관 공간 없음 → 수신자가 받을 때까지 송신자 대기
  - bounded capacity: 큐가 꽉 차면 송신자 대기
  - unbounded capacity: 송신자는 절대 기다리지 않음
- **직접 통신** (프로세스 이름 지정) vs **간접 통신** (메일박스/포트 경유)
- **blocking(동기)** vs **non-blocking(비동기)** send/receive

## 두 모델 비교
| | 공유 메모리 | 메시지 전달 |
|---|---|---|
| 속도 | 빠름 (커널 개입 최소) | 상대적으로 느림 (시스템 콜 기반, 커널 개입) |
| 적합한 경우 | 대량 데이터 공유 | 적은 양의 데이터, 분산 시스템 |
| 동기화 | 프로그래머 책임 | 충돌 회피 불필요 |

## IPC 시스템 예시
- POSIX 공유 메모리, Mach 메시지 전달, Windows ALPC
- **파이프(Pipe)**
  - 일반(익명) 파이프: **단방향** → 양방향 통신에는 파이프 2개 필요
  - 전형적 사용: 부모가 파이프 생성 → `fork()`한 자식과 생산자-소비자 방식으로 통신
  - **named pipe**: 이름이 있어 부모-자식 관계가 아니어도 통신 가능, 양방향 지원

# 3.8 클라이언트-서버 시스템에서의 통신

## 소켓 (Socket)
- 통신의 endpoint, `IP 주소 + 포트 번호`로 식별
- 클라이언트-서버가 서로의 소켓 주소를 알면 TCP/UDP로 통신
- 같은 컴퓨터 내 프로세스 간에도 소켓 통신 가능 (loopback)
- 구조화되지 않은 **바이트 스트림**만 주고받는 저수준 방식

## RPC (Remote Procedure Call)
- 네트워크로 연결된 시스템 간 **프로시저 호출을 추상화** → 원격 함수를 로컬 함수처럼 호출
- **Stub**: 클라이언트/서버 양쪽에 존재하며 호출·반환 과정을 대신 처리하는 코드 조각
- **Marshalling / Unmarshalling**: 서로 다른 주소 공간 때문에 매개변수를 메시지로 변환/역변환하는 과정 (엔디안 등 데이터 표현 차이도 흡수)
- 호출 흐름:
  1. 클라이언트가 매개변수를 클라이언트 stub에 전달
  2. stub이 매개변수를 마샬링해 서버로 전송
  3. 서버 stub이 언마샬링 후 실제 프로시저 호출
  4. 결과를 마샬링해 클라이언트로 반환 → 클라이언트 stub이 언마샬링해 호출자에게 전달
- **IDL**(Interface Definition Language)로 인터페이스를 정의하면 stub 코드가 자동 생성됨 (예: gRPC의 `.proto`)
- 호출 의미론: 메시지 유실 대비 → 타임스탬프 기반 **at most once**, ACK + 재전송을 더한 **exactly once**
- 현대적 예: Java RMI, gRPC
