## 프로세스

프로그램이 실행어 메모리에 로드되면, 프로세스가 된다. 프로세스의 메모리 구조는 스택, 힙, 데이터, 텍스트 영역으로 나뉘어져 있다.

스택은 함수 호출시의 매개변수, 복귀주소를 저장한다. 힙은 프로그램이 실행되며 동적으로 할당되는 메모리를 저장한다. 스택과 힙은 프로그램이 실행되면서 크기가 동적으로 변한다. 하지만 둘이 겹치면 안되므로, 서로 반대의 방향으로 크기가 커지도록 설계되어 있다.

- 스택
    - 함수를 호출할 때 임시로 저장해야 하는 데이터(매개변수, 복귀 주소, 지역 변수 등)를 저장
- 힙
    - 프로그램 실행 중에 동적으로 할당되는 메모리(예: C언어의 `malloc`, Java의 `new`)를 저장하는 구역
- 데이터
    - 전역 변수
- 텍스트
    - 코드 자체가 들어감.

</aside>

### 프로세스 상태

프로세스의 상태는 현재 활동에 따라서 달라진다. 

프로세스가 I/O 작업을 해야 한다고 가정해보자. 프로세스는 I/O 작업을 요청한 직후엔, 대기 상태가 된다. 이벤트를 기다린다. I/O 작업이 완료 되었다는 인터럽트가 오면, **바로 실행상태로 돌아가는 게 아니라 준비 상태로 바뀐다.** 그리고 **스케줄러 디스패치에 의해서 선택되면 다시 실행 상태**가 된다.

<aside>
❓

실행 중인 프로세스가 I/O 작업을 요청한 직후, 다시 자신의 작업을 수행하기까지 이 프로세스는 어떤 상태를 거쳐서 실행 상태로 돌아올까?

</aside>

### PCB

각 프로세스는 자신의 고유한 데이터인 PCB를 가지고 있다. (PCB는 커널이 관리한다.) 

PCB가 컨텍스트다. 따라서 컨텍스트 스위치 시, 현재 프로세스의 CPU 레지스터 값, 프로세스 상태, 메모리 관리 등을 PCB에 저장하고, 새롭게 실행되어야 할 프로세스의 정보를 담고 있는 PCB에서 정보를 꺼내서 실행한다. 문맥 교환이 진행되는 동안 CPU는 유의미한 작업을 하지 못하고 대기하므로, 이 교환 시간은 시스템 입장에서 순수한 오버헤드(시간 낭비)가 된다.

PCB에는 다음의 정보들이 담겨 있다.

- Process state: 프로세스의 상태.
- Program counter: 해당 프로세스가 이어서 실행해야 할 명령의 주소를 가리키는 카운터.
- CPU registers: 프로세스가 인터럽트 이후 올바르게 작업을 이어가기 위해 참조하는 CPU 레지스터 값.
- CPU-scheduling information: 프로세스의 중요도, 스케줄링 큐 포인터 등 스케줄링 파라미터 정보.
- Memory-management information: base, limit 레지스터 값, 페이지 테이블 등 메모리 시스템 정보.
- Accounting information: 사용된 CPU 총량, 프로세스 개수, 시간 제한 등.
- I/O status information: 프로세스에 할당된 입출력 장치 목록, 열린 파일 목록 등.

CPU 코어를 다른 프로세스로 교환할 때(문맥 교환), 커널은 현재 프로세스의 '무엇(어떤 정보들)'을 ‘어디에’ 저장하고 복원할까?

### 스케줄링 큐

프로세스가 시스템에 들어오면 잡 큐에 들어간다. 레디 큐에는 메인 메모리에서 실행을 기다리는 ready 상태의 프로세스들이 있다.

## 프로세스 간 통신 (IPC)

프로세스 간 통신 방법은 메시지 전달 방식과 공유 메모리 방식이 있다. 

메시지 패싱 방식은 우편과 같다. 

- **send와 receive 같은 시스템 콜을 사용하여 커널 모드로 전환하여** 상대에게 신호를 주고받는다.
- 커널이 통신을 관리해주므로 안전하고 동기화가 비교적 쉽다.
- 매번 커널을 거쳐야하므로(시스템 콜 오버해드) 컨텍스트 스위칭이 발생하기 때문에 속도가 느리다.
- 따라서 분산 시스템에서 주로 사용하는 방식이다.

반면 공유 메모리 방식은 게시판과 같다. 

- **OS에 공유되어 있는 메모리**에서 신호를 주고 받으면 되어 속도가 빠르다.
- 그러나 동시에 데이터를 수정하다 발생하는 충돌을 관리할 필요가 있어 프로그래머가 직접 동기화를 챙겨야 한다. (락 같은 동기화 기법 필요) 즉, 구현 난이도가 높다.

<aside>
❓

메시지 패싱 방식, 공유 메모리 방식 두 방식 중 공유 메모리 방식이 속도가 더 빠른 이유와, “왜” 프로그래머 입장에서 구현하기 더 까다로운지 설명하라.

</aside>

#### Producer-Consumer Problem

협력하는 프로세스 중 정보를 생산하는 프로세스를 생산자(Producer), 정보를 소비하는 프로세스를 소비자(Consumer)라고 부른다. 생산자-소비자 문제는 **두 프로세스가 동시에 동작할 때 일어나는 이슈**를 말한다. 보통 정보가 생산되는 속도가 소비하는 속도보다 빠르기 때문에 동기화 문제가 발생하는데, 이를 해결하기 위해 생산된 데이터를 담아두는 버퍼(Buffer)를 사용한다. 크기에 한계가 있는 버퍼를 유한 버퍼(Bounded buffer), 버퍼의 시작과 끝을 이어붙여 크기가 무한한 버퍼를 무한 버퍼(Unbounded buffer)라고 한다.

#### Synchronization

메시지 패싱의 동기화 문제를 해결하기 위해 blocking 방식과 non-blocking 방식이 사용된다.

- Blocking send: 수신자가 메시지를 받을 때까지 송신자는 block된다.
- Blocking receive: 메시지를 수신할 때까지 수신자는 block된다.
- Non-blocking send: 송신자가 메시지를 보내고 작업을 계속한다.
- None-blocking receive: 수신자가 유효한 메시지나 Null 메시지를 받는다.

## 프로세스 생성

### **fork와 exec**

- **`fork`:** 새로운 자식 프로세스를 생성한다. (현재 프로세스의 복제본이 생성된다)
    - 이때 자식 프로세스에는 '0'이 반환되고, 부모 프로세스에는 새로 생성된 자식의 'pid(0보다 큰 정수)'가 반환되어 서로를 구분할 수 있다.
    - 자식 생성 후에 부모는 노는게 아니라, fork() 함수 호출 이후의 남은 코드를 계속해서 실행해 나가고, 자식 역시 자식대로 할당된 코드를 실행한다. 하나의 작업 흐름이 두 갈래로 나뉘어 동시에 흘러간다고 생각하면 된다.
    - 따라서 두 프로세스는 동시에 작동한다.
- **`exec`:** 현재의 메모리 공간을 새로운 이진 파일(프로그램)로 덮어쓴다.
    - 부모와 똑같은 작업을 하는게 아니다.

### **좀비 프로세스와 고아 프로세스**

- **좀비(Zombie) 프로세스**
    - 실행을 종료했지만 부모 프로세스가 아직 `wait()` 시스템 콜을 호출하지 않아, 종료 상태가 프로세스 테이블에 계속 남아있는 프로세스.

- **고아(Orphan) 프로세스:** 부모 프로세스가 자식보다 먼저 종료(exit)해 버려서 남겨진 자식 프로세스

좀비, 고아 프로세스가 문제가 되는 이유 : PID, PCB 등이 회수되지 않고 메모리(커널)에 남아있기 때문.

<aside>
❓

만약 **자식 프로세스는 실행을 마치고 종료되었는데, 부모 프로세스가 바쁘거나 오류가 생겨서 아직 `wait()` 시스템 콜을 호출하지 않고 있다면** 어떻게 될까?

</aside>
