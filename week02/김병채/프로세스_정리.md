# 운영체제 Chapter 3. 프로세스 (Process) 요약

---

## 3.1 프로세스의 개념

### 프로세스란?
- **프로세스 = 실행 중인 프로그램**
- 프로그램은 디스크에 저장된 **수동적(Passive)** 존재, 프로세스는 **능동적(Active)** 존재
- 프로세스는 실행 위치를 나타내는 **프로그램 카운터(PC)** 를 가짐
- 실행 파일이 메모리에 로드되는 순간 프로그램은 프로세스가 됨
- 같은 프로그램으로 여러 개의 프로세스가 존재할 수 있음 (예: 브라우저 창 여러 개)

### 프로세스의 메모리 구조
| 영역 | 내용 | 크기 |
|---|---|---|
| Text | 실행 코드 | 고정 |
| Data | 전역 변수 | 고정 |
| Heap | 실행 중 동적 할당되는 메모리 | 가변 |
| Stack | 함수 매개변수, 복귀 주소, 지역 변수 등 임시 데이터 | 가변 |

### 프로세스 상태 (5가지)
```
New → Ready ⇄ Running → Terminated
              ↑      ↓
              ← Waiting
```
- **New**: 생성 중
- **Ready**: CPU 할당을 기다리는 상태
- **Running**: 명령어 실행 중
- **Waiting**: I/O 완료 등 이벤트를 기다리는 상태
- **Terminated**: 실행 종료
- Running 중 I/O가 필요하면 wait queue에서 대기 → I/O 완료 시 ready queue로 이동 → 다시 Running

### PCB (Process Control Block)
프로세스마다 커널이 유지하는 정보 저장소. 프로세스를 중단했다가 재시작하기 위한 데이터를 담는다.
- 프로세스 상태 (new, ready, running, waiting 등)
- 프로그램 카운터: 다음에 실행할 명령어의 주소
- CPU 레지스터: 재스케줄 시 이어서 실행하기 위한 레지스터 상태
- CPU 스케줄링 정보: 우선순위, 스케줄링 큐 포인터 등
- 메모리 관리 정보: 기준/한계 레지스터, 페이지 테이블 등
- Accounting 정보: CPU 사용량, 시간 제한 등
- I/O 상태 정보: 프로세스에 할당된 I/O 장치 목록 등

> 단일 스레드 프로세스는 프로그램 카운터 1개, 멀티스레드 프로세스는 스레드당 1개씩 가짐

## 3.2 프로세스 스케줄링

- **목적**: CPU 코어를 프로세스 간에 자주 전환해 CPU 사용 효율을 최대화하고, 사용자가 여러 프로그램과 계속 상호작용할 수 있게 함
- 프로세스들은 **ready queue**(CPU 대기)와 **wait queue**(이벤트 대기)를 오가며 관리됨
- **CPU 스케줄러**: 레디 큐의 프로세스를 선택해 CPU 코어에 할당. 정해진 시간이 지나면 프로세스를 내리고 다른 프로세스를 올림
- **스와핑(swapping)**: 메모리 부담을 줄이기 위해 프로세스를 아예 메모리에서 내리는 방식. 이후 최근 상태 그대로 다시 로드됨

### 문맥 교환 (Context Switch)
- 현재 프로세스의 상태(context)를 PCB에 **저장**하고, 다음 프로세스의 PCB에서 상태를 **복원**하는 작업
- 교환 중에는 유용한 작업을 하지 못하므로 **순수 오버헤드(pure overhead)**

## 3.3 프로세스에 대한 연산 (생성과 종료)

### 프로세스 생성
- 각 프로세스는 고유한 **pid**(process identifier)로 식별
- 부모-자식 관계로 **트리**를 형성하며, pid 1인 `systemd`(또는 `init`)가 최상위 루트 프로세스
- **fork()**: 부모의 복제본인 자식 프로세스 생성 (프로그램 카운터까지 복제 → 생성된 시점의 코드부터 실행)
- **exec()**: 새 프로그램을 현재 프로세스의 주소 공간에 덮어씌움
- **wait()**: 부모가 자식의 종료를 기다림
- fork()의 반환값: 부모에게는 자식의 pid, 자식에게는 0 → 반환값으로 부모/자식 분기 가능

### 프로세스 종료
- 마지막 명령 후 `exit()`으로 OS에 삭제 요청 → 부모에게 상태 값 반환, 자원은 OS에 반환
- 부모가 자식을 강제 종료(kill)하는 경우:
    - 자식이 할당된 자원 이상을 사용할 때
    - 자식의 작업이 더 이상 필요 없을 때
    - 부모가 종료되었고 OS가 자식의 계속 실행을 허용하지 않을 때 (연쇄 종료, cascading termination)

### 좀비 & 고아 프로세스
- **좀비(zombie)**: 자식은 종료됐지만 부모가 아직 `wait()`를 호출하지 않은 상태
- **고아(orphan)**: 부모가 `wait()` 없이 먼저 종료된 경우 → `init`/`systemd`가 새 부모가 되어 수거

## 3.4~3.7 프로세스 간 통신 (IPC)

협력하는 프로세스가 통신하는 이유: 정보 공유, 계산 가속, 모듈성 등

### 두 가지 기본 모델

#### ① 공유 메모리 (Shared Memory)
- 두 프로세스가 공유 메모리 영역을 설정하고 직접 읽고 씀
- 대표 예: **생산자-소비자 문제** (bounded buffer)
- 커널 개입이 적어 **빠름**, 대신 동기화는 프로그래머 책임

#### ② 메시지 전달 (Message Passing)
- `send()` / `receive()` 연산으로 통신
- **버퍼링(큐) 방식 3가지**
    - zero capacity: 보관 공간 없음 → 수신자가 받을 때까지 송신자 대기
    - bounded capacity: 큐가 꽉 차면 송신자 대기
    - unbounded capacity: 송신자는 절대 기다리지 않음
- **직접 통신** (프로세스 이름 지정) vs **간접 통신** (메일박스/포트 경유)
- **blocking(동기)** vs **non-blocking(비동기)** send/receive

#### 두 모델 비교
| | 공유 메모리 | 메시지 전달 |
|---|---|---|
| 속도 | 빠름 (커널 개입 최소) | 상대적으로 느림 (시스템 콜 기반, 커널 개입) |
| 적합한 경우 | 대량 데이터 공유 | 적은 양의 데이터, 분산 시스템 |
| 동기화 | 프로그래머 책임 | 충돌 회피 불필요 |

### IPC 시스템 예시
- POSIX 공유 메모리, Mach 메시지 전달, Windows ALPC
- **파이프(Pipe)**
    - 일반(익명) 파이프: **단방향** → 양방향 통신에는 파이프 2개 필요
    - 전형적 사용: 부모가 파이프 생성 → `fork()`한 자식과 생산자-소비자 방식으로 통신
    - **named pipe**: 이름이 있어 부모-자식 관계가 아니어도 통신 가능, 양방향 지원

## 3.8 클라이언트-서버 시스템에서의 통신

### 소켓 (Socket)
- 통신의 **끝점(endpoint)**, `IP 주소 + 포트 번호`로 식별
- 클라이언트-서버가 서로의 소켓 주소를 알면 TCP/UDP로 통신
- 같은 컴퓨터 내 프로세스 간에도 소켓 통신 가능 (loopback)
- 구조화되지 않은 **바이트 스트림**만 주고받는 저수준 방식

### RPC (Remote Procedure Call)
- 네트워크로 연결된 시스템 간 **프로시저 호출을 추상화** → 원격 함수를 로컬 함수처럼 호출
- **Stub**: 클라이언트/서버 양쪽에 존재하며 호출·반환 과정을 대신 처리하는 코드 조각
- **Marshalling / Unmarshalling**: 서로 다른 주소 공간 때문에 매개변수를 메시지로 변환/역변환하는 과정 (엔디안 등 데이터 표현 차이도 흡수)
- 호출 흐름:
    1. 클라이언트가 매개변수를 클라이언트 stub에 전달
    2. stub이 매개변수를 마샬링해 서버로 전송
    3. 서버 stub이 언마샬링 후 실제 프로시저 호출
    4. 결과를 마샬링해 클라이언트로 반환 → 클라이언트 stub이 언마샬링해 호출자에게 전달
- **IDL**(Interface Definition Language)로 인터페이스를 정의하면 stub 코드가 자동 생성됨 (예: gRPC의 `.proto`)
- 호출 의미론: 메시지 유실 대비 → 타임스탬프 기반 **at most once**, ACK + 재전송을 더한 **exactly once**
- 현대적 예: Java RMI, gRPC
